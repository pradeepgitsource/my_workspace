# Enterprise Architecture Guide\n\n## Domain-Driven Design Implementation\n\nThis application follows Domain-Driven Design (DDD) principles with clear business domain separation and enterprise-grade architecture patterns.\n\n## Business Domains\n\n### 1. Booking Domain (`app/booking/`)\n**Responsibility**: Managing flight reservations and seat assignments\n\n- **Repository**: `BookingRepository` - Data access for bookings\n- **Service**: `BookingService` - Business logic for booking operations\n- **Controller**: `BookingController` - HTTP endpoints for booking management\n\n### 2. Check-in Domain (`app/checkin/`)\n**Responsibility**: Web check-in process and boarding pass generation\n\n- **Repository**: `CheckinRepository` - Data access for check-in records\n- **Service**: `CheckinService` - Business logic for check-in process\n- **Controller**: `CheckinController` - HTTP endpoints for check-in operations\n\n### 3. Flight Domain (`app/flight/`)\n**Responsibility**: Flight information and seat availability\n\n- **Repository**: `FlightRepository` - Data access for flight information\n\n### 4. Passenger Domain (`app/passenger/`)\n**Responsibility**: Passenger information management\n\n- **Repository**: `PassengerRepository` - Data access for passenger data\n\n### 5. Shared Domain (`app/shared/`)\n**Responsibility**: Cross-cutting concerns and common functionality\n\n## SOLID Principles Implementation\n\n### Single Responsibility Principle (SRP)\n- Each class has one reason to change\n- Repositories handle only data access\n- Services contain only business logic\n- Controllers manage only HTTP concerns\n\n### Dependency Inversion Principle (DIP)\n- High-level modules don't depend on low-level modules\n- Both depend on abstractions (interfaces)\n\n## Exception Handling Strategy\n\n### Business Exceptions\nAll business exceptions inherit from `BaseBusinessException`:\n\n```python\nclass BaseBusinessException(Exception):\n    def __init__(self, message: str, status_code: int):\n        self.message = message\n        self.status_code = status_code\n```\n\n### Global Exception Handling\nCentralized exception handlers provide consistent error responses:\n\n```json\n{\n  \"error\": true,\n  \"message\": \"Booking not found\",\n  \"type\": \"BookingNotFoundError\",\n  \"path\": \"/api/bookings/123\"\n}\n```\n\n## Testing Strategy\n\n### Domain-Aligned Testing\nTests are organized by business domain with 95%+ coverage requirement.\n\n### Test Quality Standards\n- **Isolation**: Proper mocking with AsyncMock\n- **Scenarios**: Success, failure, and edge cases\n- **Business Logic**: Validation and transaction testing
